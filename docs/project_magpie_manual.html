Magpie Agent Manual

이 문서는 현재 워크스페이스의 실제 코드 경로를 기준으로 작성되었습니다. 사용자 실행 엔트리포인트는 magpie_agent/run.py, 시그널 감시 엔트리포인트는 bat_daemon/run.py입니다. 요청에 언급된 daemon/bat_daemon.py는 현재 저장소에는 없고, 실제 BatDaemon 구현은 bat_daemon/run.py 안에 있습니다.

Entry Point A magpie_agent/run.py 텔레그램 메시지를 받아 Owl/Hawk/Meerkat 그래프를 실행
Entry Point B bat_daemon/run.py 업비트 1시간 캔들을 실시간 감시하고 신호 도달 시 직접 체결 후 EXPIRED 타점만 다시 갱신
Shared DB The Nest strategies, monitoring_targets, wallets
Agent Chain Owl → Hawk → Meerkat Daemon은 직접 체결하고 EXPIRED 타점 재계산 시에만 refresh 그래프를 호출

공통 그래프

Owl Director

  • 도구 호출이 없으면 바로 종료하고 텔레그램으로 일반 답변을 보냅니다.
  • transfer_to_agent(hawk_picker)이면 Hawk로 넘어갑니다.
  • transfer_to_agent(meerkat_scanner)이면 Meerkat로 바로 넘어갑니다.
  • 그 외 register_strategy_to_nest, get_my_active_strategy, get_wallet, process_trade_execution는 Owl Tools에서 실행되고, 완료 후 다시 Owl로 돌아갑니다.

Hawk Picker

  • Phase 1: 전략을 읽어 후보 코인을 고르고 store_hawk_candidates를 호출하면 meerkat_mode="chart_only"가 설정됩니다.
  • Phase 2: Meerkat 차트 분석 결과를 바탕으로 update_strategy_target_coins를 호출하고 다시 Meerkat full timing 단계로 넘깁니다.

Meerkat Scanner

  • chart_only면 DB 쓰기 없이 차트 리포트만 만들고 Hawk에 되돌려 최종 종목을 고르게 합니다.
  • full timing이면 전략, 현재 지갑, 최근 매매 기록, 기존 타점 정보를 모두 읽고 최종 타점과 buy_allocation_pct를 계산한 뒤 register_monitoring_targets_to_nest를 실행합니다.

BatDaemon 조건 분기

Direct Execution + Refresh

감시 루프

  • sync_targets_from_db()는 60초마다 monitoring_targets를 재조회합니다.
  • 감시 코인 목록이 바뀌면 현재 웹소켓 연결을 닫아 새 목록으로 재구독합니다.
  • watching_coins가 비어 있으면 웹소켓을 열지 않고 같은 간격으로 대기합니다.

실시간 BUY / SELL

  • 상태가 HOLDING이면 현재가가 익절가 이상일 때 SELL, 손절가 이하일 때 SELL 시그널이 발생합니다.
  • 상태가 WAITING_BUY이고 trigger_basis == TOUCH이며 현재가가 매수 허용 구간 안에 있으면 BUY 시그널이 발생합니다.

CLOSE 기반 BUY

  • 새 1시간 캔들이 시작될 때 직전 캔들을 마감 캔들로 판정합니다.
  • 상태가 WAITING_BUY, 기준이 CLOSE, 종가가 범위 내, 거래량이 최소 기준 이상, 필요 시 양봉 마감이면 BUY 시그널이 발생합니다.

시그널 발생 후

  • BUY/SELL 시그널이 나오면 Owl을 거치지 않고 Daemon이 즉시 지갑 체결을 수행합니다.
  • BUY는 monitoring target의 buy_allocation_pct만큼 현재 원화 잔고를 사용해 체결합니다.
  • dry_run이면 BUY는 HOLDING, SELL은 EXPIRED로 메모리상 백테스트 처리만 진행합니다.
  • 매수 완료 후 상태는 HOLDING이 되고, 매도 완료 후 상태는 EXPIRED가 됩니다.
  • DB에 EXPIRED 타점이 보이면 Daemon이 Meerkat refresh 그래프를 호출해 새 WAITING_BUY 타점을 다시 계산합니다.

그래프 구조와 사용처

LangGraph 분리 현황

magpie_agent/graph.py는 현재 build_common_graph()의 alias입니다. 실제 그래프 빌더는 magpie_agent/graphs/ 아래에 있고, shared.py가 Owl/Hawk/Meerkat 노드와 공통 edge 조립을 담당합니다.

Common: 사용자 대화 진입점 Target Refresh: Daemon 후속 계산 Signal Trigger: 현재 미사용에 가까운 경량 경로 Daily Report: 정기 점검 확장용

그래프 분리 기준

용도별 조립
  • shared.py는 그래프 자체가 아니라 노드 추가 함수와 conditional edge 조립 함수 모음입니다. 그래프를 새로 나눌 때 우선 여기 재사용 가능 여부를 봐야 합니다.
  • 사용자 대화, Daemon refresh, 정기 점검처럼 진입 목적이 다르면 별도 builder로 분리하고, 상태 플래그(from_daemon, is_daily_review)와 포함 노드를 최소화하는 편이 안전합니다.
  • 현재 실사용 기준으로는 Common Graph와 Target Refresh Graph가 핵심이고, Signal Trigger와 Daily Report는 준비된 확장 그래프 성격이 더 강합니다.
  • Owl 프롬프트 선택은 magpie_agent/agents/owl_director/node.py에서 is_daily_review → from_daemon → user 순서로 갈립니다.
  • 그래프 분리 작업 시에는 import 위치뿐 아니라 실제 호출 경로가 남아 있는지도 같이 확인해야 문서와 코드가 어긋나지 않습니다.

그래프 시각화

builder별 요약 다이어그램

아래 다이어그램은 각 builder가 어떤 노드만 묶고, 어디서 끝나는지 빠르게 비교할 수 있게 축약한 뷰입니다.

common.py / build_common_graph()

사용자 인터랙션 기본 그래프. Owl → Hawk → Meerkat 전체 수직 플로우를 모두 포함합니다.

사용처: magpie_agent/run.py, dashboard/session.py, magpie_agent/graph.py
상태 플래그: 보통 from_daemon=False, is_daily_review=False

target_refresh.py / build_target_refresh_graph()

SELL 완료 후 EXPIRED 타점을 다시 WAITING_BUY 후보로 재계산할 때 쓰는 Meerkat 중심 그래프입니다.

사용처: bat_daemon/run.py에서 refresh graph 생성, bat_daemon/integrations/target_refresh.py에서 invoke
특징: Owl/Hawk 없이 바로 Meerkat로 시작하고 저장 후 종료

signal_trigger.py / build_signal_trigger_graph()

원래는 Daemon 시그널 도달 시 Owl이 먼저 검증하고 필요시 Meerkat로 이어지는 경량 그래프용으로 만들어졌습니다.

노드 구성: Owl Director, Owl Tools, Meerkat Scanner, Meerkat Tools
사용처: 현재 저장소 내 직접 호출 지점 없음

daily_report.py / build_daily_report_graph()

공통 그래프와 구조는 같지만 Owl이 daily prompt를 쓰도록 is_daily_review를 전제로 하는 정기 점검용 그래프입니다.

노드 구성: Common Graph와 동일
사용처: 현재 저장소 내 직접 호출 지점 없음

현재 운영 경로에서 중요한 그래프

build_common_graph()는 사용자 대화 진입점의 실질적인 메인 그래프이고, build_target_refresh_graph()는 Daemon 후속 재계산 경로의 핵심입니다.

현재는 대기 중인 그래프

build_signal_trigger_graph(), build_daily_report_graph()는 설계상 의미는 있지만 현재 엔트리포인트에 직접 연결되지는 않습니다.

엔티티 관계

누가 무엇을 읽고 쓰는가
  • strategies: Owl이 저장/수정, Hawk가 조회 및 target_coins 갱신, Meerkat이 조회합니다.
  • monitoring_targets: Meerkat이 최종 타점과 buy_allocation_pct를 저장하고, BatDaemon은 WAITING_BUY/HOLDING만 감시합니다. EXPIRED는 감시에서 제외됩니다.
  • wallets: Owl Tools도 조회/수정할 수 있지만, 현재 BatDaemon도 BUY/SELL 시 직접 수정합니다. 지갑에는 자산 외에 전체 체결 이력인 trade_history도 함께 저장됩니다.
  • BatDaemon: 조건을 만족하면 즉시 체결하고, 매도 완료 후 EXPIRED 타점을 다시 Meerkat에 맡겨 갱신합니다.

조건 매트릭스

핵심 if/route

Owl

tool call이 없으면 그래프를 종료하고 텔레그램에 일반 답변을 보냅니다.

Owl Tools

register_strategy_to_nest가 끝나면 Hawk Picker가 자동 호출됩니다.

Hawk Phase 1

store_hawk_candidates 호출 후 Meerkat이 차트 분석 전용 단계로 진입합니다.

Meerkat chart_only

meerkat_mode == "chart_only"이면 리포트만 만들고 Hawk로 복귀합니다.

Meerkat full timing

전략, 지갑, 최근 매매 기록, 기존 타점을 반영해 최종 타점과 매수 비율을 계산하고 DB에 저장합니다.

BatDaemon HOLDING

status == HOLDING이면 익절가와 손절가를 기준으로 SELL을 체크합니다.

BatDaemon TOUCH BUY

WAITING_BUY + TOUCH + 가격범위 충족이면 즉시 BUY 시그널이 발생하고 buy_allocation_pct 비율만큼 체결합니다.

BatDaemon CLOSE BUY

WAITING_BUY + CLOSE + 거래량/양봉 조건 충족이면 캔들 마감 기준 BUY가 발생합니다.

Expired Refresh Graph

EXPIRED 타점이 발견되면 Meerkat이 새 타점을 계산해 다시 WAITING_BUY로 갱신합니다.

검증 메모

코드 기준 작성
  • 문서의 엔트리포인트, 라우팅, DB 컬렉션명, 상태값은 현재 저장소의 Python 코드와 대조해 작성했습니다.
  • 특히 Daemon 쪽은 직접 체결, HOLDING/EXPIRED 상태 전환, EXPIRED refresh 그래프, CLOSE 조건 검사를 누락하지 않도록 반영했습니다.
  • Magpie Agent 쪽은 Common / Target Refresh / Signal Trigger / Daily Report 그래프를 파일 단위로 다시 대조해 현재 사용 여부까지 구분해서 표기했습니다.

추가 설명이 필요한 부분

  • Meerkat의 chart_only 단계는 DB 저장 단계가 아니라 Hawk의 최종 선정 정확도를 올리기 위한 중간 분석 단계입니다.
  • BatDaemon이 발생시키는 BUY/SELL은 이제 실제 체결로 바로 이어집니다. Owl은 이 경로의 체결 검증에 더 이상 개입하지 않습니다.
  • monitoring_targets.status는 현재 코드 기준으로 WAITING_BUY → HOLDING → EXPIRED → WAITING_BUY 루프를 따릅니다.
  • bat_daemon/backtest.py와 dashboard의 Backtest 뷰도 이 직접 체결 로직을 backtest_id 전용 DB 문서 기준으로 최대한 동일하게 재생합니다.
  • dashboard와 bat_daemon/backtest.py는 원본 전략 user_id를 읽어 백테스트 전용 backtest_id의 전략/지갑/타점을 새로 만든 뒤 과거 tick으로 재생합니다.
  • dashboard는 전역 사이드바 대신 Agent 탭과 Bat Daemon 탭 안에서 각각 필요한 입력값을 직접 받도록 정리되어 있습니다.
  • signal_trigger.py와 daily_report.py는 지금 당장 엔트리포인트에 연결되어 있지 않지만, 그래프 분리 후보로 남아 있는 구조입니다.
